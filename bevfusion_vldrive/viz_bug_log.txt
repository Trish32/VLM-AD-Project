==============================================================================
 BEVFusion visualisation (bev_viz.py) — bug log
==============================================================================
Bugs hit while building the shared 6-camera + LiDAR-BEV detection visualiser
(bev_viz.py) used by bevfusion_vl/visualize.py and BEVFusion_robust_vl/
visualize.py. Boxes are lidar-frame [x, y, z, w, l, h, yaw, vx, vy].


------------------------------------------------------------------------------
BUG 1 — BEV oriented sideways (forward not up)
------------------------------------------------------------------------------
Symptom
    In the LiDAR BEV panel the road ran left-right (horizontal) and the ego's
    forward direction pointed sideways, not up.

Root cause
    The BEV mapped lidar +x to "up". But the nuScenes LiDAR_TOP frame has
    +x = right, +y = forward (the same ~-90° lidar/ego offset seen in the
    bevformer / sparse4d / simulator projects). So "x up" puts RIGHT up and the
    forward road appears horizontal.

Fix  (bev_viz.bev_panel)
    Map +y (forward) → up, +x (right) → right:
        col = (x - lo) / rng * size        # right  → right
        row = (hi - y) / rng * size        # forward→ up
    (points and box corners use the same to_px).


------------------------------------------------------------------------------
BUG 2 — 3-D boxes rotated 90° (horizontal in BEV, misaligned on cameras)
------------------------------------------------------------------------------
Symptom
    BEV box rectangles sat perpendicular to the road; on the camera images the
    projected boxes were twisted and didn't wrap the vehicles.

Root cause
    Box corners were built with length l along the raw `yaw`. The detector's yaw
    is NOT the lidar-frame heading directly — the nuScenes eval converts it as
    `Quaternion(z, -yaw - pi/2)` (see tools/eval*.py `to_global`). So the true
    heading in the lidar frame is `-yaw - pi/2`; using `yaw` is 90° off.

Fix  (bev_viz._corners_3d)
    head = -yaw - pi/2;  length l along `head`, width w perpendicular.
    Boxes now align with travel direction in BEV and wrap vehicles on the cameras.


------------------------------------------------------------------------------
BUG 3 — the two ports use DIFFERENT box-z conventions
------------------------------------------------------------------------------
Symptom
    With a single z rule, one project's boxes were grounded but the other's
    floated / sank. Treating z as bottom grounded BEVFusion_robust_vl but lifted
    bevfusion_vl boxes by ~h/2; treating z as centre did the opposite.

Root cause — different detection heads emit different z:
    • BEVFusion_robust_vl (anchor3d head):   z = box BOTTOM.
    • bevfusion_vl       (TransFusion head): z = box CENTRE (gravity centre).

    Verified per project by measuring box z against the *local* LiDAR ground
    under each near vehicle:
        robust:  z - ground        ≈ +0.06 / -0.15 m   → z = bottom
        MIT:     z - h/2 - ground  ≈ +0.14 / -0.14 m   → z = centre
    Both `tools/eval*.py` happen to use gravity_centre = `z + h/2`; that is
    correct for the robust head but a latent (metric-invisible) over-shift for
    the MIT head — and nuScenes mAP/NDS, being 2-D, never reveal it.

Fix  (bev_viz._corners_3d / composite, ``z_center`` flag)
    Box bottom z0 = (z - h/2) if z_center else z; corners span [z0, z0+h].
    Drivers pass z_center=True for bevfusion_vl (TransFusion), False for the
    robust port (anchor3d). Both now rest on the ground.

    Lesson: do not assume one box convention across "the same" model family —
    the detection head decides whether z is centre or bottom, and a 2-D metric
    won't catch a mismatch.


------------------------------------------------------------------------------
ENVIRONMENT NOTE (not a viz bug, but blocks running the viz on MPS)
------------------------------------------------------------------------------
    BEVFusion-PP voxelization calls scatter_reduce(..., reduce='amin') on int64,
    which MPS does not support → run visualize.py / infer.py with `--device cpu`.


------------------------------------------------------------------------------
Takeaway
------------------------------------------------------------------------------
3-D detection that "evaluates fine" can still carry a height/heading convention
mismatch, because the standard nuScenes metrics are 2-D. Always sanity-check the
box convention (heading sign/offset, z = centre vs bottom) against a real 3-D
render before trusting downstream geometry.
